## Problem Statement

The existing `intuition` skill teaches agents to execute Intuition Protocol operations (create atoms, triples, deposit, read) but has no memory of how to do it. Every new agent session starts cold. Every agent that encounters a new error has to rediscover the same solutions. Every integration pattern has to be re-derived from scratch.

The goal is a supplementary skill — `intuition-learning` — that enables agents to commit code learnings to the Intuition knowledge graph as persistent, multiplayer memory. The flywheel: agents publish learnings → learnings accumulate in the graph → future agents bootstrap from the graph → faster integrations → more successful builds → more learnings published.

## What Makes This Cool

Three things at once, which is rare:

1. **Emergent knowledge graph** — the graph visibly grows and improves as agents build. You can watch it learn on the Intuition portal in real time.

2. **Zero-doc bootstrap** — an agent that has never seen Intuition before can onboard itself by reading what previous agents published, without a single human-written doc. The docs write themselves.

3. **Cross-agent memory** — agents pick up mid-conversation where a completely different user's agent left off. Not just one agent learning, but the whole network learning.

The EUREKA in this design: vault deposits are the trust signal. Learnings ranked by TRUST deposited are self-correcting — wrong patterns attract no signal and surface last, correct patterns attract deposits and surface first. This is fundamentally different from private memory (can't be corrected by others) or static docs (require human maintenance). The graph is self-healing.

## What Counts as a Learning

All four types are in scope:

- **Integration patterns** — how to call specific Intuition operations correctly (e.g., "pin to IPFS before encoding, always include all fields with empty string defaults")
- **Error → solution pairs** — specific errors and their resolutions (e.g., "Request Transformation Failed = omitted field in pin mutation, fix: include all fields with empty string")
- **Code snippets that worked** — reusable patterns and working implementations
- **Project-level context** — what app was being built, what Intuition was used for, overall intent. These are published as learnings with `learning_type → project_context` and linked to a project atom. Bootstrap queries can filter to "all learnings from projects similar to mine" — useful for cross-project knowledge transfer.

## Publish Trigger

End of task / session — agent reflects and publishes a consolidated summary of what it learned.

## Constraints

- Must remain a pure Markdown skill (no code dependencies) — same pattern as the existing `intuition` skill
- Must work with the existing Intuition Protocol V2 API (no new infrastructure)
- Published learnings must be useful to a cold agent with no prior Intuition context
- Must not require the `intuition` skill to be loaded simultaneously (standalone)

## Premises

1. **Separate skill, not an extension.** The existing skill is for executing protocol operations. This skill is for reflection and memory. Different cognitive modes — mixing them makes both worse.

2. **A learning maps to an Intuition atom** (Thing schema) with triples that categorize it:
   - `LearningAtom → "about" → ConceptAtom` (e.g., "IPFS pinning", "createTriples")
   - `LearningAtom → "project" → ProjectAtom`
   - `LearningAtom → "resolves" → ErrorPatternAtom` (when applicable)
   - `LearningAtom → "supersedes" → PriorLearningAtom` (when updating stale knowledge)

3. **Two equal modes, trust-ranked READ.** READ mode bootstraps by querying learnings sorted by vault deposit (TRUST signal = community confidence). High-deposit learnings are surfaced first. WRITE mode publishes structured learnings at session end. Neither mode works without the other.

4. **Agent-driven reflection protocol.** At session end, the skill gives the agent a structured reflection prompt to extract 1-3 publishable learnings from the session. The agent decides what's worth publishing; the skill provides the format and the criteria.

## Cross-Model Perspective

Codex (gpt-5.4) reviewed this cold. Key additions it contributed:

- **Falsifiable claims framing:** "agents publish falsifiable integration claims that other agents can verify, dispute, and rank" — turns the graph into a self-healing public runbook, not a scrapbook. Each learning should include evidence, code, chain, and timestamp. Future agents can write back `confirmed` or `superseded_by`.

- **`supersedes` triple is non-negotiable.** "Without that, the graph rots." Protocols change. V2 replaced V1. Learnings from six months ago may actively mislead.

- **The hole:** Trust boundary. "If you do not solve verification, supersession, and poisoning resistance, the whole thing collapses into permanent cargo-cult memory: one wrong pattern gets published once, and every later agent confidently repeats it." Resolution: vault-ranked bootstrap query.

- **Skip list:** no generic "remember everything," no background daemon, no staking/reputation system in v1, no free-form transcript storage, no cross-protocol memory.

## Approaches Considered

### Approach A: Thin Appendix Skill
New skill SKILL.md that delegates all actual publishing to the existing `intuition` skill. A reflection layer, not a write layer. Ships in 1-2 days. Missing: canonical ontology, supersedes logic, trust-ranked queries. A cold agent couldn't fully bootstrap from it.

### Approach B: Full Standalone Skill (chosen)
Complete self-contained skill — SKILL.md, reference files (ontology, read-queries, reflection protocol), operations (publish-learning, bootstrap), test prompts. Canonical pre-seeded concept atom IDs. `supersedes` triple. Confidence field. Dedup query. Trust-ranked bootstrap. Ships in 1 week.

### Approach C: Learning Mode Inside Existing Skill
Add a "Learning Mode" section to `skills/intuition/SKILL.md`. Zero new installs, organic adoption on next update. But mixes cognitive modes, makes the existing skill harder to follow, and can't evolve independently.

## Recommended Approach

**Approach B — Full Standalone Skill**

The vision (emergent graph, cross-agent memory, zero-doc bootstrap) requires a complete, opinionated spec. A thin appendix won't bootstrap a cold agent. A section buried in the existing skill will be ignored.

The canonical ontology — pre-seeded atom IDs for "IPFS pinning," "createAtoms," "createTriples," "pin mutation," "error handling," etc. — is the key artifact. Without it, every agent has to re-discover what concepts exist before it can find relevant learnings.

### Data Model

**LearningAtom** (Thing schema):
```json
{
  "name": "createTriples requires all atoms to be IPFS-pinned first",
  "description": "Attempting createTriples with a plain-string atom ID causes MultiVault_TermDoesNotExist. All three atoms (subject, predicate, object) must exist on-chain as IPFS-pinned atoms before calling createTriples. Fix: run createAtoms for each atom and confirm on-chain before calling createTriples. Evidence: tx 0x... on testnet. Confidence: 4/5 (verified by test run). Code: see snippet below.",
  "url": "",
  "image": ""
}
```

The `description` field follows a canonical template (all sections required; use "N/A" when not applicable):

```
## Explanation
[What the learning is. What happens if you don't follow it.]

## Fix
[Exact action to take. One sentence.]

## Evidence
[tx hash, error log, test run, or observed behavior that confirms this]

## Confidence
[N/5 — reason for rating]

## Code
[working code snippet, or N/A]
```

This template is the same for all learning types. The sections after `## Explanation` may be N/A but must be present. Consistent structure ensures bootstrap agents can read learnings from any publisher.

No machine-parsed structured fields in description — queryable data is stored as triples only.

**Queryable structured metadata is stored as triples** (all atom IDs from ontology):

| Triple | Purpose | Queryable |
|--------|---------|-----------|
| `learning → about → concept_atom` | What operation/concept this learning relates to (e.g., createTriples) | Yes — by concept_atom_id |
| `learning → learning_type → type_atom` | Category: integration_pattern, error_solution, code_snippet, project_context | Yes — by type_atom_id |
| `learning → project → project_atom` | Which project produced this learning | Yes — by project_atom_id |
| `learning → resolves → error_atom` | Error pattern this resolves (name = exact error string) | Yes — by error_atom_id or label |
| `learning → supersedes → prior_learning_atom` | Replaces a stale/incorrect prior learning | Yes — by prior_learning_atom_id |

**ErrorAtom** (Thing schema) — created on first occurrence, reused after:
```json
{
  "name": "MultiVault_TermDoesNotExist",
  "description": "Error returned when createTriples references an atom ID that does not exist on-chain.",
  "url": "",
  "image": ""
}
```

Dedup check for errors: before publishing an error_solution learning, query for existing ErrorAtom with `label = exact error string`. If found, reuse the atom ID and check if an existing learning already resolves it. If learning exists with that error_atom_id as object of `resolves` predicate, skip or supersede.

**Confidence rubric** (stored in description text, not as a queryable field):
- **1/5** — guessed or assumed, not verified
- **2/5** — tried once and it worked (single observation)
- **3/5** — applied in two or more separate tasks
- **4/5** — confirmed by test run, CI pass, or explicit tx hash
- **5/5** — independently verified by a second agent or human

### Trust-Ranked Bootstrap Query Pattern

```graphql
query BootstrapLearnings($conceptId: numeric!, $minShares: numeric!) {
  triples(
    where: {
      # ABOUT_PREDICATE_ID: hardcoded constant from ontology.md (not a runtime variable)
      predicate_id: { _eq: <ABOUT_PREDICATE_ID_FROM_ONTOLOGY> }
      object_id: { _eq: $conceptId }
      vault: { total_shares: { _gte: $minShares } }
    }
    order_by: { vault: { total_shares: desc } }
    limit: 10
  ) {
    subject {
      id
      label
      description
    }
    vault {
      total_shares
    }
  }
}
```

**Trust metric:** `total_shares` = number of share tokens held across all depositors. Higher total shares = more TRUST deposited by the community = higher confidence in this learning. Not to be confused with `current_share_price` (price per share, not participation breadth).

**`$minShares` defaults:**
- Testnet / cold graph: `0` (accept all learnings — nothing has vault deposits yet)
- Mainnet with history: `100` (recommended starting threshold — tune based on graph activity)

An agent starting on a cold graph (zero learnings published yet) will get empty results. This is expected. The skill should handle this gracefully: "No learnings found in graph. Proceeding without bootstrap context — publish learnings after this session to seed the graph."

**Trust signal bootstrapping (v1 minimum viable):** A correct learning with zero vault deposits ranks the same as a wrong one on a fresh graph. To ensure ranking is non-trivial from day one: the publishing agent should deposit a nominal amount (1 tTRUST on testnet, 1 TRUST on mainnet) on its own learning at publish time. This gives every published learning a baseline signal. The deposit is a weak signal (self-attested) but it differentiates published-and-self-verified from published-with-zero-confidence. Instructions for the deposit step belong in `operations/publish-learning.md`.

### Reflection Protocol (session end)

The skill prompts the agent with 5 questions:

1. What Intuition operations did you attempt this session?
2. Did you encounter any errors? What were they and how were they resolved?
3. What patterns did you find yourself applying repeatedly?
4. What code or query worked that you'd want to reuse?
5. What would you tell a cold agent starting this same task tomorrow?

For each answer that produces a publishable learning:

1. **Dedup check:** Query the graph for learnings already linked to the same concept_atom AND (for error_solution type) the same error_atom. This is a graph traversal, not a text search — exact match on atom IDs. If a matching learning already exists, evaluate whether yours supersedes it (adds new evidence, corrects it) or is redundant. If redundant, skip publishing.
   - **Capture the prior learning's atom ID** from `subject.id` in the query result. Store it as `prior_learning_atom_id`. If you decide to supersede, use this ID in the `supersedes` triple: `new_learning_atom → supersedes → prior_learning_atom_id`.

2. **Assign confidence** using the 5-point rubric in the data model section.

3. **Publish** via the operations/publish-learning.md flow.

## Open Questions

1. **Canonical ontology seeding ownership.** The `ontology.md` file lists atom IDs that must exist on-chain before the skill works. On testnet, these should be seeded by the Intuition team using the existing `intuition` skill (create atoms via `skills/intuition/operations/create-atoms.md`). On mainnet, check if atoms already exist before creating (use GraphQL label search). **Owner: Intuition team (Jonathan or Billy). Blocker for all other work.** Ontology.md should include both testnet and mainnet IDs — two columns.

2. **Dedup:** Resolved — dedup is a graph traversal, not text search. Same concept_atom + same error_atom (for error_solution type) = duplicate. If no error_atom applies, same concept_atom + same learning_type_atom = evaluate for supersession (no time window — the graph has no queryable timestamp; use vault deposit as proxy for "canonical" status: vault deposit > 0 = treat as potentially canonical, evaluate for supersession; vault = 0 = safe to supersede or skip). Client-side logic in reflection-protocol.md.

3. **Confidence calibration:** Resolved — 5-point rubric defined in data model section.

4. **supersedes trigger:** An agent should check for learnings to supersede in two cases: (a) when it discovers a prior learning is wrong based on current behavior, or (b) when publishing a learning about the same concept_atom and operation as an existing learning, and the new learning has higher confidence or more recent evidence. The agent does not need to know the protocol version changed — it supersedes based on contradictory evidence.

5. **`confirmed`/`superseded_by` write-back:** Scoped out of v1. Future versions may add a confirmation predicate so agents can signal "I ran this and it worked." Not required for the bootstrap flywheel to function.

## Success Criteria

- A cold agent loading `intuition-learning` can bootstrap from the graph and correctly execute a `createAtoms` + `createTriples` flow without reading the main `intuition` skill first
- An agent that encounters `Request Transformation Failed` during a pin mutation can query the graph and find the resolution (include all fields with empty string defaults) with vault deposit > 0
- An agent building a project similar to a prior project can query project-context learnings and get useful framing for their integration work
- After a session, a well-run reflection produces 1-3 atoms with correct triples on testnet
- The published learnings are visible on the Intuition portal and make sense to a human reading them
- Bootstrap output is formatted as a numbered list of learning summaries (name + confidence + one-sentence fix) that an agent can apply immediately without reading the full description
