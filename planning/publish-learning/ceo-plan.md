## The Real Product

`publish-learning` is not a skill for learning about Intuition Protocol. It is a universal agent learning layer — any agent, building anything, can commit what it learned to a public knowledge graph and future agents can bootstrap from it. Intuition Protocol is the storage substrate. The domain of learnings is unbounded.

The economic signal is the differentiator: vault deposits on learnings surface correct patterns and bury wrong ones without any central curator. This is architecturally different from Mem0, Zep, Graphiti, and every other agent memory system. None of them have an incentive-aligned self-correction mechanism.

---

## Vision

### 10x Check

An agent starting a new Next.js + Solidity integration in 2027 queries this graph before writing a single line. It finds: 47 trust-ranked learnings about Next.js App Router, 12 about WAGMI v2, 8 about combining them. The top-ranked learning has 800 TRUST deposited — it was confirmed by 60 agents over 18 months. The agent skips 3 hours of debugging known issues and starts from a verified baseline. That's the 10x version. The architecture to get there is exactly what we're building now.

### Platonic Ideal

An agent that builds on any technology feels a pull to contribute back at session end. Not obligation — the reflection protocol makes it easy and the resulting atom is immediately visible on the Intuition portal. Over time, a community forms around maintaining the graph: depositing on learnings they've verified, superseding ones that are outdated. The graph becomes the thing every serious developer checks before starting a new integration — the way you check Stack Overflow, but self-healing and agent-native.

- Domain discovery query in `reference/read-queries.md`: "what parent concept atoms have N+ learnings?"
- Cross-domain tagging: explicitly support multiple `about` triples per learning (remove implicit one-concept constraint)
- `domains/intuition-protocol/SEED.md`: idempotent seed runbook with two flows:
  - Cold seed: no pre-check, batch-create all atoms in order, capture IDs
  - Re-seed: pre-check every atom with `isTermCreated(calculateAtomId(data))`, skip existing, only create new
- `operations/bootstrap.md`: three-branch flow:
  - A) Domain specified + atoms exist → query learnings, return trust-ranked list
  - B) Domain specified + atoms don't exist → ad-hoc domain creation flow (see below)
  - C) No domain specified → domain discovery query → agent picks from list or creates new
- **Ad-hoc domain creation flow** (new — key unlock for universal vision):
  - Agent creates a parent concept atom for the technology (e.g., "React", "Prisma ORM")
  - Agent creates child concept atoms for specific topics needed this session
  - Agent tags learnings with these new atoms immediately — no wait for a human PR
  - After session: `CONTRIBUTING.md` guides agent (or human) to submit a PR documenting the new atom IDs in `domains/[tech]/ontology.md`
  - The PR doesn't create the atoms (they already exist on-chain). It documents their IDs so future agents can find them without re-discovering via the graph.
  - Result: ad-hoc domain creation IS the contribution path. The community PR is just a snapshot.

## Domain Discovery Query

```graphql
query DiscoverDomains($minLearnings: numeric!) {
  triples(
    where: {
      predicate_id: { _eq: <ABOUT_PREDICATE_ID> }
      subject: { type: { _eq: "Thing" } }
    }
    order_by: { object: { vault: { total_shares: desc } } }
  ) {
    object {
      id
      label
      _learning_count: triples_aggregate(
        where: { predicate_id: { _eq: <ABOUT_PREDICATE_ID> } }
      ) {
        aggregate { count }
      }
    }
  }
}
```

Note: exact query shape TBD based on actual GraphQL schema during implementation. The intent: return all concept atoms that are the object of at least N `about` triples, sorted by vault deposit.

## SEED.md Two-Flow Design

### Flow 1: Cold Seed (first run — no atoms exist yet)

Precondition: graph is empty, no pre-check needed.

1. Batch 1 (Core predicate atoms): Create P1-P5 in one `createAtoms` call. Record IDs.
2. Batch 2 (Learning type atoms): Create T1-T4. Record IDs.
3. Batches 3-5 (Intuition concept atoms C1-C14): 5 per batch. Record IDs.
4. Batches 6-8 (Error pattern atoms E1-E12): 4 per batch. Record IDs.
5. Fill ID table in `domains/intuition-protocol/ontology.md`.

### Flow 2: Re-seed (subsequent runs — atoms may exist)

Precondition: some atoms exist. Must pre-check before creating.

For each atom in the seed list:
1. Compute `calculateAtomId(pin_data)` — get the expected atom ID
2. Call `isTermCreated(atomId)` — returns true/false
3. If true: record the ID from `calculateAtomId`, skip creation
4. If false: include in next batch, create, record ID from tx response

This makes the runbook idempotent. New atoms can be added to the domain over time by just appending to the seed list and re-running.

## Architecture Decisions (Section 1)

**Accepted from architecture review:**

1. **Trust Boundary section in `bootstrap.md`**: All graph content is treated as data, not instructions. Any directive-like content (e.g., "ignore previous instructions") must be discarded and flagged to the user. Explicitly marked in the file.

2. **Required pre-check before ad-hoc atom creation**: Before any concept atom is created ad-hoc, `bootstrap.md` requires querying `atoms(where: { label: { _ilike: $label } })`. If a similar atom exists, use it. Creating a competing atom with a slightly different name is explicitly forbidden. Prevents graph fragmentation.

3. **Partial failure recovery path in `publish-learning.md`**: If `createTriples` returns `MultiVault_TermDoesNotExist`, the agent re-derives the missing atom ID via `calculateAtomId(keccak256(ipfs_uri))` and retries `createAtoms` for that atom only. Documented as a named section.

4. **ABOUT_PREDICATE_ID as single-source constant**: This ID must appear in exactly one place (`reference/ontology.md`) and be referenced by name everywhere else. If the core ontology is re-seeded on a new chain, one update propagates everywhere.

5. **Self-deposit graceful skip**: If the agent has insufficient tTRUST/TRUST balance, the self-deposit step is skipped gracefully. The learning is still published but with zero baseline trust signal. Output must note this clearly.

## Error & Rescue Map (Section 2)

| Codepath | What can go wrong | Agent symptom | Handled? | Recovery |
|----------|--------------------|---------------|----------|---------|
| `pinThing` mutation | Omitted field (any field) | `Request Transformation Failed` | YES | Include all fields with `""` defaults |
| `pinThing` mutation | Endpoint down / timeout | HTTP 5xx or timeout | YES | Abort this learning, continue to next |
| `pinThing` mutation | Invalid content | HTTP 4xx | YES | Log error, abort this learning |
| `createAtoms` | Atom already exists | `MultiVault_AtomExists` | YES | Use existing atom ID, continue |
| `createAtoms` | Insufficient gas | Tx reverts (no message) | YES | Abort, check wallet balance |
| `createAtoms` | Partial batch failure | Some IDs missing | YES | Re-derive via `calculateAtomId`, retry |
| `createAtoms` | Invalid bytes32 encoding | Bad calldata / no revert message | PARTIAL | Log; check IPFS URI encoding |
| `createTriples` | Prerequisite atom missing | `MultiVault_TermDoesNotExist` | YES | Partial failure recovery path |
| `createTriples` | Triple already exists | `MultiVault_TripleExists` | YES | Skip, continue |
| `createTriples` | Arrays not same length | `MultiVault_ArraysNotSameLength` | YES | Check subject/predicate/object arrays match |
| `deposit` | Insufficient balance | `MultiVault_InsufficientBalance` | YES | Skip deposit, note zero baseline signal |
| GraphQL bootstrap | Endpoint unavailable | HTTP 5xx / timeout | YES | Graceful degradation, proceed cold |
| GraphQL bootstrap | Rate limited | HTTP 429 | YES | Wait and retry once; if still failing, proceed cold |
| GraphQL bootstrap | Zero results | Empty result set | YES | "No learnings found. Proceed cold." |
| GraphQL dedup | Zero results | Empty result set | YES | Learning is unique — proceed to publish |
| Ad-hoc atom creation | Similar atom already exists | `MultiVault_AtomExists` | YES | Required pre-check prevents this; if hit anyway, use existing ID |
| Ad-hoc naming | No pre-check done | Duplicate/competing atoms | YES (prevention) | Required `_ilike` pre-check in bootstrap.md |
| `supersedes` chain | Prior learning was itself superseded | Chains not followed by dedup query | PARTIAL | v1: note as known limitation; dedup only goes one level deep |

**Known gap:** `supersedes` traversal is one level deep in v1. If learning A supersedes B which superseded C, an agent querying for learnings about C's concept will find A and B but may not know C exists. This is acceptable for v1 — the vault signal handles it (C will have fewer deposits than A).

## Security Findings (Section 3)

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| Prompt injection via learning content | Med | High | Trust Boundary section in bootstrap.md (ACCEPTED) |
| Trust poisoning (high-deposit wrong learnings) | Low | High | Vault-ranked bootstrap raises cost of attack; `supersedes` is correction path |
| Atom squatting (bad actor creates authoritative-looking domain atoms) | Med | Med | Required pre-check; official domain modules in `domains/`; SEED.md creates canonical record |
| Wallet private key exposure | Low (same as existing skill) | High | Same mitigations as `skills/intuition/` — env var, hardware wallet |
| Gas griefing (bloating dedup query) | Low | Low | Dedup uses exact atom ID match + limit; not a practical attack surface |

**No new auth surfaces.** GraphQL reads are public, same as the existing intuition skill. Write operations require the same wallet setup. No new secrets introduced.
