## What This Document Is

This is the atom catalog and seeding spec for `skills/intuition-learning/reference/ontology.md`. It defines exactly which atoms need to exist on-chain before the `intuition-learning` skill can function. Once seeded, atom IDs from this document get committed to `ontology.md` as hardcoded constants.

This document is the source of truth for the ontology. `ontology.md` in the skill is derived from this.

## Design Decisions

Four decisions made in session:

1. **Natural language names** — atoms use "Create Atoms", "IPFS Pinning", not `createAtoms` or `create-atoms`. Matches how they appear in the Intuition portal. No mental translation needed when browsing the graph.

2. **One IPFS Pinning atom** — `pinThing`, `pinPerson`, `pinOrganization` all map to a single `IPFS Pinning` concept atom. Schema-specific details live in the learning description. Keeps queries simple; avoids 3x duplication of shared learnings.

3. **Create all predicate atoms fresh** — no searching for existing "about" atoms. We own the predicate semantics. Avoids ambiguity about which community atom is canonical for this skill.

4. **Intuition Protocol parent atom** — every learning gets two `About` triples: one specific (e.g., `About → Create Atoms`), one general (`About → Intuition Protocol`). Cold agents can bootstrap with a single broad query before they know what operation to ask about.

## Complete Atom Catalog

All are `Thing` schema. All descriptions follow the pattern established here. (Atom count: see ID table at end — do not hardcode here as it changes when atoms are added.)

### Predicate Atoms (5)

Used in the triple-predicate position to form relationships between learnings and metadata.

| # | Name | Description | Schema |
|---|------|-------------|--------|
| P1 | About | Indicates that a learning relates to the specified concept, operation, or topic. Used to connect a LearningAtom to a ConceptAtom. | Thing |
| P2 | Learning Type | Categorizes a learning into one of four types: Integration Pattern, Error Solution, Code Snippet, or Project Context. | Thing |
| P3 | Resolves | Indicates that a learning provides the solution to a specific error pattern. Used to connect a LearningAtom to an ErrorPatternAtom. Only used with T2 (Error Solution) learnings — do not use with T1, T3, or T4. | Thing |
| P4 | Supersedes | Indicates that a learning replaces or corrects a prior learning. Used when new evidence contradicts or improves upon an existing published learning. | Thing |
| P5 | Project | Links a learning to the project that produced it. Used to connect a LearningAtom to a ProjectAtom. | Thing |

### Learning Type Atoms (4)

Used as objects of the `Learning Type` predicate. Every learning atom must have exactly one `Learning Type` triple.

| # | Name | Description | Schema |
|---|------|-------------|--------|
| T1 | Integration Pattern | A learning that describes the correct way to call an Intuition Protocol operation, including required parameters, ordering, and common pitfalls. | Thing |
| T2 | Error Solution | A learning that documents a specific error encountered during Intuition Protocol integration and the verified fix for that error. | Thing |
| T3 | Code Snippet | A learning that contains working code (query, mutation, transaction encoding) that can be directly reused in an Intuition integration. | Thing |
| T4 | Project Context | A learning that describes the overall purpose of a project, how Intuition was integrated into it, and the key architectural decisions made. | Thing |

### Error Pattern Atoms

Used as objects of the `Resolves` predicate. Split into two subcategories with different naming conventions:

#### Contract Revert Errors (E1–E6)
Names use the exact revert string returned by the MultiVault contract. Agents see these in `cast call` simulation output or transaction failure messages. Exact-match lookup: query by exact name.

| # | Name | Description | Schema |
|---|------|-------------|--------|
| E1 | MultiVault_TermDoesNotExist | Contract revert when createTriples references an atom ID that does not exist on-chain. All three atoms (subject, predicate, object) must be created before calling createTriples. | Thing |
| E2 | MultiVault_AtomExists | Contract revert when createAtoms is called with data that maps to an atom that already exists. Use the existing atom ID instead of creating a new one. | Thing |
| E3 | MultiVault_TripleExists | Contract revert when createTriples is called with a subject-predicate-object combination that already exists. Use the existing triple ID instead. | Thing |
| E4 | MultiVault_InsufficientBalance | Contract revert when msg.value does not equal the sum of assets[] in a batch operation. Recalculate total value. | Thing |
| E5 | MultiVault_InsufficientAssets | Contract revert when assets[i] is less than the minimum creation cost for that operation. Query atomCost or tripleCost and use the returned value. | Thing |
| E6 | MultiVault_ArraysNotSameLength | Contract revert when parallel input arrays (e.g. atoms[] and assets[]) have different lengths. Ensure all batch arrays have the same element count. | Thing |

#### Client-Side / Off-Chain Errors (E7–E12)
Names use readable descriptions — these errors don't have a single canonical machine string. Agents match by recognizing the pattern, not exact string.

| # | Name | Description | Schema |
|---|------|-------------|--------|
| E7 | Request Transformation Failed | GraphQL error returned when a pin mutation is missing a required field. All fields in pinThing, pinPerson, and pinOrganization must be present, using empty string for optional fields. Exact message: "Request Transformation Failed". | Thing |
| E8 | Pin Failed: Missing URI | IPFS pin returned success but no valid ipfs:// URI was in the response. Retry the pin mutation. Do not proceed to createAtoms without a valid URI. | Thing |
| E9 | Pin Failed: HTTP Error | IPFS pin returned a non-2xx HTTP status. Check network connectivity and GraphQL endpoint. Retry up to 3 times before surfacing as a pin_failed error. | Thing |
| E10 | Transaction Revert: No Message | Transaction simulation reverts with no message. Usually indicates ABI encoding mismatch or unrecognized function selector. Verify bytes32 IDs and curveId parameter. | Thing |
| E11 | GraphQL Auth Error | GraphQL endpoint returned 401 Unauthorized. Check that the correct endpoint URL is being used and that no auth headers are being sent (the Intuition GraphQL API requires no authentication). | Thing |
| E12 | GraphQL Rate Limit | GraphQL endpoint returned 429 Too Many Requests. Add exponential backoff between queries. Reduce pagination batch size. | Thing |

## Project Atoms (Not Seeded — Created Per-Project)

Project atoms are not part of the core seeded ontology. They are created ad-hoc by the agent at the start of each session using the standard `pinThing` + `createAtoms` flow.

**Required structure for a ProjectAtom:**
```json
{
  "name": "[Project Name]",
  "description": "Project built using Intuition Protocol. [1-2 sentences on what the project does and how it uses Intuition.] Chain: [mainnet|testnet]. Agent session: [date].",
  "url": "[repo URL or empty string]",
  "image": ""
}
```

Before creating a project atom, the agent should search for an existing atom matching the project name (`atoms(label: "[Project Name]")`). If found, reuse the existing ID rather than creating a duplicate.

## Usage Rules

### Every learning atom must have:
1. `Learning Type` triple (exactly one) → T1, T2, T3, or T4
2. `About → Intuition Protocol` triple (C1) — always
3. `About → [specific concept]` triple (C2–C13) — at least one
4. `Project` triple → project atom for the session
5. `Resolves` triple → error atom (E1–E12) — only for T2 (Error Solution) type

### Optional:
6. `Supersedes` triple → prior learning atom — when this learning corrects or improves a prior one

### Query patterns:

**Bootstrap before a specific operation** (e.g., about to call createTriples):
```
Filter: About predicate → C3 (Create Triples)
Order: vault.total_shares desc
Limit: 5
```

**Cold bootstrap** (no idea what operation, just starting out):
```
Filter: About predicate → C1 (Intuition Protocol)
Order: vault.total_shares desc
Limit: 10
```

**Error lookup** (just saw MultiVault_TermDoesNotExist):
```
Filter: Resolves predicate → E1 (MultiVault_TermDoesNotExist)
Order: vault.total_shares desc
Limit: 3
```

**Supersedes traversal** (check if a learning has been superseded):
```
Filter: Supersedes predicate, subject = [learning_atom_id]
→ returns the newer learning that supersedes this one
(if result is non-empty, prefer the superseding learning)
```

**Supersedes discovery** (check if you should supersede before publishing):
```
Filter: About predicate → [concept_atom_id] + Learning Type → T2
Order: vault.total_shares desc
Limit: 5
→ review result descriptions; if your new learning contradicts one,
  record its subject.id as prior_learning_atom_id for the Supersedes triple
```

## Seeding Process

**Who:** Intuition team (Jonathan or Billy). Required before any agent can use the skill.

**When:** Testnet first. Mainnet only after testnet is validated.

**How:** Use the existing `skills/intuition` skill to create all 32 atoms in batch.

### Order

Predicates and types must be created before any learnings reference them, but for seeding purposes order within each group doesn't matter. Recommended order:

1. Create all 5 predicate atoms (P1–P5) — `createAtoms` batch
2. Create all 4 learning type atoms (T1–T4) — `createAtoms` batch
3. Create the parent concept atom (C1: Intuition Protocol) — `createAtoms` single
4. Create remaining concept atoms (C2–C13) — `createAtoms` batch
5. Create all error pattern atoms (E1–E12) — `createAtoms` batch

Each batch uses `pinThing` + `createAtoms`. No triples are created during seeding — these atoms are building blocks for learnings, not for expressing relationships between themselves.

**Batch size:** Recommended max 10 atoms per `createAtoms` call to stay well within gas limits. The seeding steps above have at most 13 atoms (step 4) ��� split step 4 into two batches of 6-7 if needed.

**Wallet config:** A funded wallet on the Intuition L3 is required. Use `cast` with `--private-key` or a viem `walletClient`. For wallet infrastructure details, see the Prerequisites section of `skills/intuition/SKILL.md`. For RPC endpoints and network config, see `skills/intuition/reference/reading-state.md`. No API keys required — only $TRUST (mainnet) or tTRUST (testnet) for gas.

### Recording IDs

**ID capture when batch has existing atoms:** Before calling `createAtoms`, run `isTermCreated(calculateAtomId(data))` for each atom. Skip existing atoms — don't include them in the batch. Capture the returned IDs from the batch call in order. Then use `calculateAtomId` to look up IDs for atoms that were skipped (already existed). This ensures you have the correct ID for every atom regardless of whether it was newly created or pre-existing.

## Open Questions

1. **IPFS Pinning schema type for error atoms:** Error atoms have technical names like `MultiVault_TermDoesNotExist`. When pinning these as Thing atoms, the `description` field explains the error in plain English. The `name` field uses the exact technical string. Confirm before mainnet seeding that the portal displays the `name` field as the primary label.

2. **Atom deduplication during seeding:** Resolved — see ID capture instructions in Seeding Process. Run `isTermCreated(calculateAtomId(data))` per atom before batching.

3. **Version field in descriptions:** Resolved — add `(Intuition Protocol V2, chainId 1155/13579)` to C1 (Intuition Protocol) description only. All other concept atoms inherit the version context from C1 via the `About → Intuition Protocol` triple that every learning carries. No need to duplicate version on every atom.

## What I noticed about how you think

- You picked "Natural language" immediately for naming — you were thinking about how atoms look in the portal, not how they map to code. That's product thinking, not engineering thinking. The ontology will be more useful for it.

- "Search first, reuse if found" was the more technically elegant choice. You picked ownership instead. That's the right call for v1 — ambiguity about which 'about' atom is canonical would have caused silent bugs in the learning queries.

- The Intuition Protocol parent atom was approved without hesitation. You already had the mental model of cold agents needing a broad entry point. That's the same thinking that drove the original bootstrap concept.
