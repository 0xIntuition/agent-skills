## Problem Statement

The Intuition knowledge graph has no shared ontology governance. Different developers and agents write atoms for the same concepts independently — "type" and "kind" and "category" all end up as separate predicates with no connection. Once two communities have fragmented onto incompatible schemas, the damage is hard to undo: existing triples reference the old atoms, and neither community knows the other's exist.

The existing skills (`intuition` and `intuition-learning`) solve the write layer and the memory layer. Neither solves the schema layer. Every new skill author has to design ontologies from scratch, manually check the graph for existing patterns, and hope other builders have done the same.

The ontology curator skill intercepts fragmentation at creation time. It takes a use case, queries the live graph, applies decision rules, detects conflicts, and emits a seed manifest — the actual createAtoms and createTriples calldata — before any new atoms hit the chain.

## What Makes This Cool

Three things at once:

1. **AI as standards committee** — no humans needed to ratify a schema. The live graph and vault deposits handle governance. An agent proposes an ontology; the graph tells it what's already canonical; TRUST tells it how much the community believes in each atom.

2. **Compiler not curator** — the user writes in their own domain vocabulary ("credential issuer", "verification result"). The skill compiles that into graph-compatible atoms, mapping each term to `reuse`, `alias`, `broader_than`, `narrower_than`, or `conflicts_with` against what already exists. No prior Intuition knowledge required.

3. **Cross-ontology conflict detection** — when two communities have used different atoms for the same concept, the skill surfaces the conflict and proposes a unification strategy: either redirect to the higher-TRUST atom or create a bridge triple. Fragmentation becomes visible before it compounds.

The EUREKA: everyone assumes ontology governance requires human committees because static spec documents can't express consensus algorithmically. Intuition's vault model breaks that assumption. TRUST deposits ARE governance. A predicate becomes canonical not by committee ratification but by the community putting money behind it.

## Constraints

- Pure markdown skill (no code dependencies) — same pattern as `skills/intuition` and `skills/intuition-learning`
- Must work for a cold agent AND a cold human with zero Intuition prior knowledge
- Must not require `intuition` or `intuition-learning` to be loaded simultaneously (standalone)
- Produces unsigned tx params — no wallet/signing management
- Works with existing Intuition Protocol V2 API (no new infrastructure needed)

## Premises

1. Intercept fragmentation at creation time, not after.
2. Must work for cold agent AND cold human (zero Intuition prior knowledge). The skill includes an embedded Intuition explainer loaded at the start of every session.
3. Live graph is source of truth — must query GraphQL before proposing any atom.
4. Vault deposits = canonicalization signal. Standards emerge from economic signals, not committee ratification.
5. Output is a seeded ontology, not a design doc. Skill produces createAtoms calldata and walks agent through seeding.
6. Cross-ontology conflict detection is first-class, not a nice-to-have.

## Approaches Considered

### Approach A: Thin Wrapper
A reference doc and prompt template. Structured ontology design guidance only — no live graph queries. Completeness 3/10. Doesn't prevent fragmentation; user still has to manually check the graph.

### Approach B: Full Standalone Skill — Compiler Pattern (Chosen)
Complete standalone skill at `skills/ontology-curator/`. Use case description → live graph queries → decision rules → conflict detection → seed manifest + unsigned calldata. Includes Intuition explainer. Completeness 9/10.

### Approach C: Living Registry
Protocol-level registry maintained by the skill; consumer skills query it. Right long-term vision but requires community adoption to be valuable. The live graph with high-TRUST atoms IS already the registry — this would duplicate it before there's critical mass. Completeness: 10/10 vision, 4/10 executable today. Natural evolution path from B once adoption exists.

### Bootstrap Requirement

**This skill cannot generate valid `createTriplesTx` for mapping and hierarchy triples until the curator's own meta-predicate atoms exist on-chain.** These are the predicates used in mapping triples (`same-as`, `broader-than`, `narrower-than`, `conflicts-with`) and in conflict documentation.

Before the skill can be used:
1. Resolve Open Question #1 (define the 4 meta-predicates with precise descriptions)
2. Pin and create those 4 atoms using `pinThing` + `createAtoms` on testnet, then mainnet
3. Record the resulting `bytes32` IDs in `reference/decision-rules.md` as constants (same pattern as `intuition-learning` ontology IDs)

The meta-predicates are the curator's own seed ontology. Until they exist on-chain, mapping entries in `mappings[]` will have `"bootstrapped": false` and `predicateTermId: null`. When any `bootstrapped: false` entry exists in the manifest, `createTriplesTx` is omitted entirely from the output. All other phases (discovery, decision tree, createAtomsTx for new atoms) work without this bootstrap.

### Skill Phases

**Phase 1 — Orientation (always run)**
Always load `reference/intuition-primer.md` at the start of a session. Explain what atoms and triples are in terms of the user's domain. Do not skip this step — the primer is fast (~3 sentences) and ensures cold users aren't lost. Example for a credential system: "An atom is a concept node — 'Credential Issuer' would be an atom. A triple connects three atoms: (Alice, issued, Credential-123). Every atom and triple has a vault; depositing $TRUST into it signals community confidence." Users already familiar with Intuition can skip ahead by saying "I know Intuition, skip primer."

**Phase 2 — Session Setup**
Before any writes, fetch and cache:
```
atomCost   = getAtomCost()   -- via cast call or viem readContract
tripleCost = getTripleCost()
```
Use these as `assets[i]` values when generating calldata (creation cost only). `msg.value = sum(assets[])`.

Note: `defaultCurveId` (from `getBondingCurveConfig()`) is NOT fetched here — this skill does not call `deposit` or `redeem`. If the user wants to deposit TRUST into newly seeded atoms after creation, load the `intuition` skill for that.

**Phase 3 — Use Case Intake**
Collect: domain name, 5-15 key concepts the user wants to represent, 3-5 relationships between concepts. The agent does not need to know Intuition — it just describes its domain. Cap at 15 concepts per session — for larger domains, run multiple sessions scoped by sub-domain. (15 concepts × top-10 per concept = 150 atoms → C(150,2) ≈ 11,175 pairs — still tractable.)

**Phase 4 — Live Graph Discovery**
For each concept, query GraphQL. Limit to 50 candidates per concept. Include neighborhood data in the same query to avoid N+1 round trips:
```graphql
atoms(
  where: { label: { _ilike: "%concept%" } }
  order_by: { as_predicate_triples_aggregate: { count: desc } }
  limit: 50
) {
  term_id label type
  term { vaults { total_shares curve_id } }
  as_predicate_triples_aggregate { aggregate { count } }
  as_subject_triples(limit: 10) { predicate { label } }
  as_object_triples(limit: 10) { predicate { label } }
}
```
Note: `term.vaults` is an array (one vault per bonding curve). For TRUST signal, use `term.vaults[0].total_shares` — on mainnet/testnet there is one vault per atom (curve_id "1" by default). `as_subject_triples` and `as_object_triples` as SELECT fields are confirmed valid against the live schema.
"Local neighborhood" for the ALIAS rule = the union of `as_subject_triples` and `as_object_triples` predicate labels. Fetching this inline avoids a second round trip per candidate.

Use `term.vaults[0].total_shares` from GraphQL as a non-real-time TRUST signal — acceptable for canonicalization decisions. For exact real-time vault state before executing a deposit, use on-chain `getVault(termId, curveId)`. Note the distinction in `reference/discovery.md`.

**Phase 5 — Decision Tree (per concept)**
For each user concept, apply exactly one rule. Rules are evaluated in order; stop at the first match:
```
1. REUSE:   exact label match (case-insensitive) AND non-TextObject type
            → DO NOT auto-select. Surface top-3 matching atoms sorted by term.vaults[0].total_shares desc.
              For each candidate show: label, termId, type, total_shares, and description (from IPFS
              metadata if available). Require agent/user to confirm which atom to REUSE, or decline all.
              If confirmed → use that termId, no new atom needed.
              If all declined → proceed to Rule 2.
              REASON: labels are display hints — multiple atoms can share the same label string.
              Never auto-select on label alone. Always confirm identity before reuse.

2. ALIAS:   semantic match: (a) >75% word token overlap between labels (lowercase, split on
            whitespace/hyphens) AND (b) at least 2 shared predicate labels in local neighborhood
            [local neighborhood = set of predicate atom labels appearing in as_predicate_triples
            where the candidate atom is subject or object; use the GraphQL query in reference/discovery.md]
            AND term.vaults[0].total_shares > 0
            → CREATE new atom + add mapping triple (newAtom, "same-as", existingAtom)
            NOTE: requires meta-predicate bootstrap for createTriplesTx. If not bootstrapped,
            mapping entry has bootstrapped=false and createTriplesTx is omitted.

3. BROADER/NARROWER: (candidate label tokens are a strict subset of existing atom label tokens,
                     OR existing atom label tokens are a strict subset of candidate label tokens)
                     AND term.vaults[0].total_shares > 0
                     [if subset relationship is unclear, REQUIRE user confirmation before applying
                     this rule — ask "Is [candidate] a specific type of [existing]?"]
                     Triple direction: (narrower_atom, "broader-than", broader_atom)
                     i.e., (Credential Issuer, broader-than, Issuer) means Credential Issuer is
                     the broader concept. Or inverse: (Issuer, "narrower-than", Credential Issuer).
                     Use the relationship that matches natural reading.
                     → CREATE new atom + add hierarchy triple

4. TEXTOBJECT: exact or semantic match exists but is TextObject type
               → SURFACE to user: "Predicate '[label]' (termId: 0x...) exists as a legacy
                 plain-string atom. Create a pinned replacement (pin via pinThing, then createAtoms).
                 New pinned version becomes canonical going forward."
               → CREATE pinned replacement atom

5. CREATE:   no match → CREATE fresh atom
```

**Phase 5b — Predicate Normalization** *(runs after Phase 5, before Phase 7)*
The fragmentation problem in live ontologies is primarily about predicates, not concepts. Two communities defining `issued-by` vs `issuedBy` vs `issuer` create silent graph partitions — triples can never be joined across them. Phase 5b resolves all predicate atoms the user's domain uses before seeding begins.

For each predicate label appearing in the user's proposed triples:
```
Normalize:  lowercase, strip hyphens/underscores, strip camelCase boundaries
            (e.g., "issuedBy" → "issuedby", "issued-by" → "issuedby")
            Use normalized form for all comparisons.

1. REUSE:   normalized label matches an existing non-TextObject atom
            → DO NOT auto-select. Surface top-3 candidates sorted by term.vaults[0].total_shares desc.
              Show: label, termId, type, total_shares, description. Require confirmation.
              If confirmed → use that termId as the predicate atom.
              If declined → proceed to Rule 2.

2. ALIAS:   normalized form is identical but surface forms differ (camelCase / kebab-case / snake_case
            of the same root, e.g., "issued-by" and "issuedBy" both normalize to "issuedby")
            → Auto-select the canonical form (highest total_shares), log normalization:
              { userLabel: "issuedBy", resolvedTermId: "0x...", normalizedForm: "issued-by", action: "ALIAS" }
              No new atom needed. This is safe to auto-apply because the forms are demonstrably the same root.

3. CREATE:  no match after normalization → CREATE fresh predicate atom via pinThing + createAtoms.

Output — add `predicates[]` block to seed manifest:
  { userLabel, resolvedTermId, action: "REUSE"|"ALIAS"|"CREATE", normalizedForm }
```

Predicate atoms created in Phase 5b enter the same `create[]` queue as concept atoms and are seeded in Phase 7 before any triples are staged. This ensures concept triples in Phase 8 can reference resolved predicate termIds.

**Phase 6 — Conflict Detection**
Scan only the candidate set returned in Phase 4. To keep comparison tractable, limit to the top-10 atoms by `term.vaults[0].total_shares` per concept — take the Phase 4 results and sort descending by `term.vaults[0].total_shares`, keep top-10 per concept. With 10 concepts, that is 100 total atoms → C(100,2) ≈ 4,950 unique pairs — manageable in-context.

Surface a conflict when:
- Two atoms in the bounded candidate set appear semantically equivalent (>50% token overlap) but have no linking triple
- A proposed CREATE atom from Phase 5 has >50% token overlap with an existing atom created by a different wallet address (check `creator` field in GraphQL if available, otherwise use no-shared-triples as proxy)
- Two atoms both appear as predicates for the same relationship type across the local neighborhoods of different concepts

For each conflict, output: `candidateA` (label + termId), `candidateB` (label + termId), `reason`, `resolution` (redirect to higher `total_shares` atom, or create unification triple using `same-as` meta-predicate).

Note: all mapping triples (`same-as`, `broader-than`, `narrower-than`, `conflicts-with`) automatically generate counter-triple vaults per Intuition Protocol Invariant 12. The semantic meaning of counter-triples for these meta-predicates is undefined in this skill — document this explicitly in `reference/conflict-patterns.md` so downstream consumers don't attempt to interpret them.

**Phase 7 — IPFS Pinning + Existence Check**
Before generating `createAtomsTx`, for each atom in `create[]`:

Step 1 — Pin to IPFS using the `pinThing` GraphQL mutation (embedded in `SKILL.md`; same mutation as `skills/intuition/reference/schemas.md`):
```graphql
mutation PinThing($input: PinThingInput!) {
  pinThing(input: $input) { uri }
}
# All fields required; use empty string for optional fields
```
Capture returned `ipfs://` URI. If pin fails, surface a `pin_failed` object (same pattern as `skills/intuition`) and halt — do not proceed to `createAtomsTx`.

Step 2 — Pre-creation existence check. Use viem `readContract` (preferred) or cast with ABI-encoded bytes. Do NOT use `cast --from-utf8` directly — it outputs raw hex, not ABI-encoded `bytes`, and will produce an incorrect termId.

**Viem (correct):**
```typescript
const atomData = toHex(ipfsUri)  // stringToHex from viem
const termId = await readContract({ address: MULTIVAULT, abi, functionName: 'calculateAtomId', args: [atomData] })
const exists = await readContract({ address: MULTIVAULT, abi, functionName: 'isTermCreated', args: [termId] })
```

**cast (correct — must ABI-encode bytes argument):**
```bash
ATOM_DATA=$(cast --from-utf8 "$IPFS_URI")
TERM_ID=$(cast call $MULTIVAULT "calculateAtomId(bytes)(bytes32)" "$ATOM_DATA" --rpc-url $RPC)
EXISTS=$(cast call $MULTIVAULT "isTermCreated(bytes32)(bool)" "$TERM_ID" --rpc-url $RPC)
```
Note: when cast takes a `bytes` argument it accepts a raw hex string from `cast --from-utf8` directly — the ABI encoding is handled by cast internally. Use viem's `toHex` (same as `stringToHex`) to produce the same value. Both paths are correct; the important thing is not to manually ABI-encode or skip the contract call.

`calculateAtomId` is a `pure` function — call it via `readContract` or `cast call`, not a local hash.

If `isTermCreated` returns `true`: atom already exists. Remove from `create[]`, add to `reuse[]` with the confirmed termId. Do not include in `createAtomsTx`.

Step 3 — Record pre-calculated `termId` in each `create[]` entry for triple encoding before on-chain confirmation.

**Phase 8 — Seed Manifest**
Emit two staged JSON objects. `createTriplesTx` MUST be submitted only after `createAtomsTx` is confirmed on-chain, because triples reference atoms that do not have on-chain existence until after creation.

Atom IDs for new atoms are deterministic and can be pre-calculated before submission: `calculateAtomId(stringToHex(ipfsUri))`. Include pre-calculated `termId` in each `create[]` entry so downstream consumers can reference them in triple encoding without waiting for on-chain confirmation.

```json
{
  "status": "proposal_ready",
  "network": "<mainnet|testnet>",
  "useCase": "<domain description>",
  "reuse": [
    {
      "label": "About",
      "termId": "0x<bytes32>",
      "reason": "high predicate usage (N triples), term.vaults[0].total_shares: M"
    }
  ],
  "create": [
    {
      "name": "Credential Issuer",
      "schema": "Thing",
      "pinInput": { "name": "Credential Issuer", "description": "...", "image": "", "url": "" },
      "ipfsUri": "ipfs://bafy...",
      "termId": "0x<calculateAtomId result>"
    }
  ],
  "mappings": [
    {
      "subject": "Credential Issuer",
      "subjectTermId": "0x<termId>",
      "predicate": "broader-than",
      "predicateTermId": "0x<meta-predicate-termId or null if not yet bootstrapped>",
      "object": "Organization",
      "objectTermId": "0x<termId>",
      "bootstrapped": true
    }
  ],
  "_note": "If bootstrapped=false on any mapping entry, createTriplesTx is omitted. Bootstrap meta-predicates first.",
  "conflicts": [
    {
      "candidateA": "Verifier",
      "candidateATermId": "0x...",
      "candidateB": "Credential Verifier",
      "candidateBTermId": "0x...",
      "reason": "same neighborhood, different labels (>50% token overlap)",
      "resolution": "reuse Verifier (higher total_shares)"
    }
  ],
  "staging": {
    "step1": "Submit createAtomsTx. Wait for confirmation.",
    "step2": "Submit createTriplesTx only after step1 confirms on-chain."
  },
  "createAtomsTx": {
    "to": "0x<MultiVault>",
    "data": "0x<calldata>",
    "value": "<wei sum of assets[]>",
    "chainId": "<1155|13579>"
  },
  "createTriplesTx": {
    "to": "0x<MultiVault>",
    "data": "0x<calldata>",
    "value": "<wei sum of assets[]>",
    "chainId": "<1155|13579>",
    "prerequisite": "createAtomsTx confirmed"
  }
}
```

**Note on deposits:** After seeding, recommend depositing a minimum of 0.001 TRUST per atom (above the creation cost) to establish a nonzero vault signal. By default the skill generates `createAtomsTx` with `assets[i] = atomCost` (creation cost only, no extra deposit). If the user wants an initial deposit, the skill asks: "Add initial deposit? Recommended minimum: 0.001 TRUST per atom. Enter amount per atom or 0 to skip." If nonzero, regenerate calldata with `assets[i] = atomCost + (initialDeposit * 1e18)` and `msg.value = sum(assets[])`. No other regeneration flows exist.

`SKILL.md` must embed the following ABI fragments so the skill works without loading `skills/intuition`. Copy from `skills/intuition/SKILL.md` ABI section — do not link to it:

Also embed the network configuration table (chain IDs, MultiVault addresses, RPC endpoints, GraphQL endpoints) from `skills/intuition/SKILL.md`.

### What This Skill Does NOT Do
- No wallet/signing management (same as existing skills)
- Does not manage a separate registry (the live graph IS the registry)
- Does not maintain a canonical list of all ontologies (each domain's atoms speak for themselves)
- No background daemon or auto-publish behavior
- No full-graph semantic scan (conflict detection is bounded to the candidate set from Phase 4)

Should use the natural language naming convention from the learning skill. Resolve before implementation — these atoms must exist on-chain before `createTriplesTx` can be generated for mappings.

## Resolved Decisions

- **TextObject handling UX:** When Phase 5 surfaces a TextObject atom, the skill outputs one sentence: "Predicate '[label]' (termId: 0x...) exists as a legacy plain-string atom. Create a pinned replacement using pinThing. The TextObject version will not be used going forward." Flow continues after user acknowledges.

## Success Criteria

- A developer with no Intuition knowledge can run the skill, describe their domain, and get a complete seed manifest in one session
- The skill prevents at least one fragmentation event by surfacing an existing canonical atom instead of creating a duplicate
- Conflict detection finds at least one real cross-community conflict on the live graph during testing (run b2 prompts against testnet)
- Seed manifest JSON is machine-readable and composable with the existing `intuition` skill's output format
- `createAtomsTx` + `createTriplesTx` are staged correctly — submitting them out of order produces `MultiVault_TermDoesNotExist`, which b3 test prompts validate
- `isTermCreated` pre-check runs before every `createAtomsTx` — re-running the skill on a partially-seeded state does not produce `MultiVault_AtomExists` reverts
- Test prompts in `tests/prompts/` cover all three phases with defined expected output shapes

## Reviewer Concerns

Issues flagged by adversarial spec review that require resolution before or during implementation. Not blockers for design approval, but an engineer implementing from this doc should address these.

1. **`getVault` ABI fragment missing from embedded list** (Completeness) — Phase 4 mentions on-chain vault reads before deposit but `getVault` isn't in the embedded ABI list. Either add it or explicitly note real-time vault reads are deferred to the `intuition` skill.

2. **`pinThing` mutation fields not inlined** (Completeness) — `PinThingInput` fields are shown by example in the JSON (`name`, `description`, `image`, `url`) but no authoritative schema definition is given. Inline the full mutation in `SKILL.md` or `operations/seed-ontology.md` during implementation.

3. **GraphQL error path undefined** (Completeness) — Phase 4 has no behavior when the GraphQL endpoint is unreachable. Define during implementation: propose all concepts as CREATE with a manifest warning, or surface `discovery_failed` and halt.

4. **ALIAS rule conditions should be labeled (a), (b), (c)** (Clarity) — the `term.vaults[0].total_shares > 0` check is syntactically ambiguous. Renumber to three explicit conditions during implementation in `reference/decision-rules.md`.

5. **`value` field should be base-10 string** (Consistency) — the seed manifest example shows `"value": "<wei sum of assets[]>"`. Clarify in `reference/output-contract.md` that this is a base-10 string, matching the existing `intuition` skill's output contract.
