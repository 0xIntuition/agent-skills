# Runtime Enforcement: Block Signing Until Validation Passes

This reference turns policy guidance into a blocking runtime pattern. The
shipped skill includes the schemas and policy examples needed to define the
contract, but you must implement the validator and signer wrapper in your own
execution environment.

## Goal

No transaction is signed unless `validateTxFromIntent` returns `status = pass`.

## Shipped Artifacts

- Schemas:
  - `reference/schemas/intent.schema.json`
  - `reference/schemas/unsigned-tx.schema.json`
  - `reference/schemas/approval-request.schema.json`
- Strict policy example: `reference/autonomous-policy.example.json`

## Executor-Owned Components

Implement these in your signer or executor pipeline:

- a validator that checks `{intent, unsigned tx, policy, runtime context}`
- a blocking wrapper that runs validation before signing or broadcasting

## Required Inputs

- `intent.json` generated by planner/policy layer (no prebuilt tx fields)
- `unsigned-tx.json` generated by transaction builder
- `autonomous-policy.json` loaded by executor
- Runtime context:
  - `rpc` URL
  - `from` signer address
  - optionally `spentTodayWei` and `pendingCount` from your state store

## Blocking Validator Step

```bash
your-validator \
  --intent /path/intent.json \
  --tx /path/unsigned-tx.json \
  --policy /path/autonomous-policy.json \
  --rpc https://rpc.intuition.systems/http \
  --from 0xYourSignerAddress \
  --spent-today-wei 0 \
  --pending-count 0
```

If this command exits non-zero, do not sign.

## One-Step Blocking Sign Command

Use the wrapper below to guarantee signing runs only after validator pass:

```bash
your-enforce-and-sign-wrapper \
  --intent /path/intent.json \
  --tx /path/unsigned-tx.json \
  --policy /path/autonomous-policy.json \
  --rpc https://rpc.intuition.systems/http \
  --from 0xYourSignerAddress \
  --spent-today-wei 0 \
  --pending-count 0 \
  -- your-signer-command --tx-file /path/unsigned-tx.json
```

Wrapper exit codes:

- `0`: validation pass and signer command succeeded
- `1`: validation failed/error or signer command failed
- `2`: approval required (blocked, signer command not executed)

## Executor Integration Pattern

1. Planner emits intent.
2. Tx builder emits unsigned tx `{to,data,value,chainId}`.
3. Executor runs validator command above.
4. If `status=pass`, sign and broadcast.
5. If fail:
   - in `manual-review` mode, return approval request
   - otherwise hard-block and alert

Using one blocking wrapper makes this flow a single required entrypoint.

## Minimum Blocking Checks

- strict tx schema (`to,data,value,chainId` only)
- chain/address allowlist match
- selector match
- calldata exact match to intent-reencoded calldata
- non-zero receiver validation for receiver-bearing operations
- value match + max value limits
- optional on-chain existence binding checks for terms/atoms
- simulation pass (when required by policy)

## Operational Notes

- Keep signer isolated from untrusted research content.
- Treat intent and policy as trusted control-plane inputs.
- Log validator result with:
  - policy version/checksum
  - `calldata` hash
  - per-check pass/fail flags

## Recommended Next Step

Call your validator and signer wrapper from signer middleware so all code paths
(API, queue worker, cron, CLI) hit the same blocking gate.
